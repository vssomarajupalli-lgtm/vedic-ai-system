# ANSWER COMPOSER ARCHITECTURE v1

## 1. Responsibilities
The Answer Composer is the final presentation layer of the Vedic AI System. Its singular responsibility is to transform the strictly constrained `FormulaEvaluationResult` into human-readable, grammatically cohesive, and culturally appropriate astrological prose.
- It translates boolean truths into natural language.
- It presents astrological evidence without calculating it.
- It enforces tonal boundaries dictated by the Formula Evaluator.

## 2. Input Contract
The Answer Composer consumes **only** the `FormulaEvaluationResult` generated by the `FormulaEvaluator`.
```python
class FormulaEvaluationResult(BaseModel):
    final_state: str  # FAVORABLE, MIXED, or UNFAVORABLE
    isolated_signals: Dict[str, Any]
    answer_template_key: str
    system_warnings: List[str]
```
The raw `ChartProcessResponse` from the mathematical engines is explicitly blocked from reaching this layer.

## 3. Output Contract
The Answer Composer outputs a `ComposerResponse` containing:
- `answer_text`: The LLM-generated string formatted in Markdown.
- `referenced_yogas`: An optional array of Yoga names isolated from the `isolated_signals` to render UI badges.

## 4. Narrative Construction Flow
1. **Template Retrieval:** The Composer looks up the `answer_template_key` mapping (e.g., `timing_assessment_v1_favorable`) based on the `final_state`.
2. **Context Injection:** The Composer constructs the LLM System Prompt using the retrieved template.
3. **Evidence Injection:** The `isolated_signals` JSON is stringified and injected into the "EVIDENCE" block of the prompt.
4. **LLM Execution:** The generative model is invoked with strict temperature settings (e.g., Temperature = 0.1) to prioritize formatting over creativity.
5. **Post-Processing:** The output is validated, and any `system_warnings` are appended.

## 5. Evidence Presentation Rules
- The Composer must only cite planetary positions or dignities explicitly provided in the `isolated_signals`.
- If the `isolated_signals` payload specifies "Venus is in 7th House," the Composer may format this as "The presence of Venus in your 7th house indicates..."
- The Composer is explicitly prohibited from inferring unprovided aspects (e.g., "Because Venus is here, Mars must be opposite...").

## 6. Confidence Layer Presentation
- If the `final_state` is `FAVORABLE`, the Composer's tone must be affirming and supportive.
- If the `final_state` is `MIXED`, the Composer must explicitly use words like "delay," "effort required," or "mixed results."
- If the `final_state` is `UNFAVORABLE`, the Composer must present the reality objectively without fatalism, focusing heavily on remedial measures or patience.

## 7. Template Resolution Model
The system maps `answer_template_key` (from the Formula Repository) + `final_state` (from the Evaluator) to a specific system prompt.
- `timing_assessment_v1` + `FAVORABLE` = Uses the `Timing_Favorable_Prompt.txt`
- `timing_assessment_v1` + `UNFAVORABLE` = Uses the `Timing_Unfavorable_Prompt.txt`
This guarantees that the LLM is forced to adopt a pre-approved rhetorical structure aligned with the mathematical truth.

## 8. Hallucination Prevention Controls
- **Zero Chart Math:** The LLM is given no birth data, longitudes, or mathematical tools.
- **Tone-Locking:** The System Prompt explicitly instructs: "The evaluation of this question is [FINAL_STATE]. You MUST conclude that the outcome is [FINAL_STATE]."
- **Bounded Evidence:** The prompt includes: "You may ONLY reference the astrological data provided in the EVIDENCE block below. Do not assume any other planets or houses exist."

## 9. Future Mandali Gochara Integration Points
When future Moon-centered Transits are activated:

- The Composer will receive specialized Gochara templates that instruct it to present these timeline arrays as bulleted chronological predictions.

## 10. ARCHITECTURAL CHALLENGE: SYSTEM WARNING PRESENTATION

**1. Documented Ambiguity:**
How should the Answer Composer handle the `system_warnings` array (e.g., missing TransitEngine data resulting in a forced `MIXED` state)? Should the LLM generate a natural language apology, or should the warning be rigidly appended?

**2. Present Alternatives:**
- *Alternative A (Generative):* Inject `system_warnings` into the LLM System Prompt with instructions: "Explain to the user that transit data is missing."
- *Alternative B (Deterministic):* The Answer Composer ignores `system_warnings` during the LLM generation phase. After the LLM returns its `answer_text`, the Answer Composer programmatically appends a hardcoded UI warning block (e.g., `> [!WARNING] Transit data unavailable.`) to the end of the text string.

**3. Present Recommendation:**
I recommend **Alternative B (Deterministic)**. 

**4. Explain Governance Implications:**
If we use Alternative A, we violate the principle of strict determinism. The LLM might hallucinate around the warning, downplay its severity, or forget to mention it entirely if the prompt context gets too long. Alternative B guarantees that engine degradation is communicated to the user with 100% mathematical certainty, free from generative interference, perfectly aligning with our anti-hallucination governance rules.
