## Calibration Layer Design
* **CalibrationManager**: The singleton provider of calibration data that deep-restores deserialized JSON into Python types (including restoring `str` keys back to `int` and `tuple` keys) and securely provisions read-only data fragments to engines.
* **Calibration Registry**: Exclusively handles loading files from the `profiles/` directory and enforces immutability.
* **Calibration Types**: Data classes establishing strict schemas for the active calibration namespace, enforcing memory segregation.
* **Calibration Profile**: The JSON file containing 100% of the numerical tuning values. 
* **Dependency Injection**: Engines are initialized via `__init__(self, calibration=None)`, dynamically creating a local `CalibrationManager` instance if omitted.

## Architecture Decisions
* **One active calibration profile**: Engines must use the exact same profile data uniformly to prevent conflicting values.
* **Read-only runtime**: Engine contracts do not permit mutation of the injected dictionary configuration.
* **No runtime tuning**: Dynamic re-weighting is banned to maintain strict consistency.
* **No formula changes**: Pure migration—only variable initialization logic changed.
* **Engine isolation**: The engines do not need to query each other; they get all math from `CalibrationManager`.

## Lessons Learned
* **JSON integer key restoration**: Dictionaries representing houses or planet IDs in Python dictionaries are corrupted into Strings natively by JSON serialization. We built `_restore_int_keys` to automatically traverse and repair them.
* **JSON tuple key restoration**: Dictionaries requiring matrix lookups (like planet-nature pairs) are similarly destroyed by JSON strings. We added targeted tuple reconstruction interceptors inside `CalibrationManager` to safeguard string-to-tuple matching.
* **Why engines were not modified**: Engine logic was kept 100% intact to satisfy immutable architecture guarantees and ensure identical downstream integration into the `PipelineRunner`.
* **Why calibration belongs outside engines**: Moving tuning data out reduces deployment drift, isolates behavioral errors, and enables robust mathematical refactoring cycles without code-level breakage.
