# Phase 15G.7A: Varga Ownership Audit

## 1. Trace of Varga Calculations

`PlanetStrengthEngine.calculate_strength()` is invoked on D9/D10 data in exactly one location across the entire backend:
- `backend/app/engines/natal_promise_engine.py` (Line 199, inside `_varga_score()`)

## 2. Caller Identification

- **Current Caller:** `NatalPromiseEngine._varga_score()`
- **Purpose:** To evaluate the domain-specific Varga chart (e.g., D9 Navamsha for Marriage, D10 Dashamamsha for Career) and derive the 15% weight pillar for the domain's primary karaka.
- **Returned Values Used:** Only `float(result.get("final_score", _NEUTRAL))` is extracted. The rest of the payload, including the full structural `breakdown` (Dignity, House, Aspects), is permanently discarded.

## 3. Governance Violation Check

**Is `NatalPromiseEngine` violating the "Zero Astrological Recalculation" rule?**
**YES.**
The architectural mandate of `NatalPromiseEngine` is to consume pre-computed scores only. However, by explicitly importing `PlanetStrengthEngine` and actively executing `calculate_strength(karaka_data, shadbala_data={})`, it functions as an active astrological orchestrator. It is bypassing the dedicated `VargaEngine` to dynamically compute Varga strengths on the fly, directly contradicting the stated immutable rule.

## 4. Single Producer Feasibility

**Can `VargaEngine` safely become the single producer of D9/D10 planet strengths, breakdowns, and modifiers?**
**YES.**
- `VargaEngine.evaluate()` already loops over every planet and has direct access to the parsed `D9` and `D10` json blobs.
- `VargaEngine` can safely invoke `PlanetStrengthEngine.calculate_strength()` for every planet in D9 and D10 natively.
- This would allow `VargaEngine` to store the *true* Varga `final_score` and the exhaustive `breakdown` object directly inside its output schema (`results["D9"]["planets"][planet_name]`), completely resolving the transparency gap identified in Phase 15G.7.

## 5. Architectural Review

### Current Architecture
- `VargaEngine` computes superficial modifiers (e.g., Vargottama bonus) but skips full structural scoring, leaving `breakdown` dictionaries empty.
- `NatalPromiseEngine` illegally calculates deep Varga structural strength, extracts a single float, and destroys the breakdown evidence.

### Recommended Architecture
- **VargaEngine (Producer):** Instantiates `PlanetStrengthEngine`. Iterates over all 9 planets for both D9 and D10. Calculates full structural strength and stores the returned `final_score` and `breakdown` natively inside `engine_outputs.vargas`.
- **NatalPromiseEngine (Consumer):** Refactored to act purely as a consumer. It will simply look up `varga_results[varga_id]["planets"][karaka]["final_score"]` with zero calculation overhead.

### Migration Risk
- **Low:** The logic executing inside `PlanetStrengthEngine` remains identical (a Varga planet dictionary with an empty shadbala dict). The scores generated by `VargaEngine` will perfectly match the scores previously generated by `NatalPromiseEngine`.

### Regression Risk
- **Low:** By relocating the calculation upstream, `NatalPromiseEngine`'s mathematical output will not shift. Furthermore, legacy outputs (`modifiers`, `confidence_flags`) generated by `VargaEngine` can be seamlessly merged with the new `breakdown` payload, ensuring no downstream dependencies are broken.
