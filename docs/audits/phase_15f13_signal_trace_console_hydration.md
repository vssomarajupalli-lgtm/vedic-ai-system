# Phase 15F.13 – Signal Trace Console Hydration

## 1. Audit of Current Signal Payload
The runtime structure of `isolated_signals` output by `SignalTranslator` has been verified. It is a flat dictionary where the keys are the semantic signals (e.g., `7th_house`, `venus`) and the values are rich dictionaries natively exported from their respective computation engines (e.g., `PlanetStrengthEngine`, `HouseStrengthEngine`).

Example Runtime Structure:
```json
{
  "venus": {
    "final_score": 63,
    "breakdown": {
      "dignity": 25.0,
      "house_placement": 10.0,
      "aspects": 5.0,
      ...
    },
    "confidence_flags": [],
    "metadata": {
      "entity_id": "venus",
      "entity_type": "planet"
    }
  }
}
```

## 2. Backend Schema Hydration
The `StructuredQuestionResult` model inside `backend/app/schemas/question.py` was extended to include the `isolated_signals` dictionary, preserving complete backward compatibility with all existing string arrays (`supporting_factors` / `attention_factors`).

**Schema Addition:**
```python
isolated_signals: Optional[Dict[str, Any]] = None
```

## 3. API Endpoint Hydration
`backend/app/formatters/display_formatter.py` was updated to map the computed `isolated_signals` generated by `SignalTranslator` into the newly created schema field of the `StructuredQuestionResult`. The `/ask-structured-question` endpoint now dynamically serves this rich trace payload alongside the standard textual commentary.

## 4. Frontend Hydration
The frontend architecture was systematically upgraded to capture and display the new traces:
- **State Management:** `frontend/src/store/useChartStore.ts` was expanded to natively capture `questionResults`.
- **Query Interception:** `frontend/src/pages/QuestionEngine.tsx` now calls `setQuestionResults` each time a structured query is answered, ensuring the most recent question’s trace data is universally available in the Zustand store.
- **Console Rendering:** The static "NOT EXPOSED" placeholder in `VerificationConsole.tsx` was replaced with a dynamic mapping block. It reads the latest structured query from the store and loops through its `isolated_signals`, hydrating an expandable UI card for each one.

## 5. UI Implementation
The Signal Trace Console now successfully implements:
- A collapsible section titled **"F. Signal Trace Console"**.
- Fallback instructions prompting the user to *"Ask a structured question in the Question Browser"* if no query has been run yet.
- Dynamic indicator showing the *"Target Formula"* currently being traced.
- One collapsible details-card per signal, displaying the **Signal Name** and its **Final Score** (color-coded).
- An expanded view revealing exact Engine **Metadata** (Entity Type, Entity ID) and a multi-column **Breakdown Grid** showing precise mathematical contributors (e.g., Dignity, Shadbala, SAV points).

## 6. Verification & Regression Assessment
- **Zero API Breakages:** The original `QuestionResultCard` remains completely unaware of the `isolated_signals` payload, continuing to render text arrays precisely as before.
- **Hydration Validation:** Queries against the 4 core domains (Marriage, Career, Wealth, Education) were tested. The backend faithfully injected the exact signal traces matching the requested domain into the UI, fully mitigating Phase 15F.10's data loss anomalies. 
- **Integrity Guarantee:** Because of the Phase 15F.12 architectural refactor, no `0/100` defaults or `ashtakavarga` overrides are present. The math is mathematically intact.
