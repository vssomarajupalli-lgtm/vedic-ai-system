# PHASE 15A: ENGINE INTERFACE AUDIT

## SECTION 1: ENGINE INVENTORY
*(Location: `backend/app/engines/`)*

*   **`ashtakavarga_engine.py` (Active):** Calculates BAV and SAV strengths. Depends on core constants. Outputs SAV/BAV charts and modifier scores.
*   **`dasha_engine.py` (Active):** Evaluates Vimshottari Dasha timing. Outputs temporal activation scores and confidence flags (e.g., `active_mahadasha`).
*   **`functional_nature_engine.py` (Active):** Determines functional benefics/malefics based on Lagna. Outputs categorized planet arrays.
*   **`house_strength_engine.py` (Active):** Evaluates Bhava Bala. Depends on AshtakavargaEngine for SAV support. Outputs 12 house objects with `final_score` and `confidence_flags`.
*   **`mandali_generator.py` (Placeholder/Legacy):** Mathematical utility for Moon-centered Padas. Outputs raw integer grid placements. Not currently integrated into the probability pipeline.
*   **`master_probability_engine.py` (Transitional/Legacy):** Calculates a weighted average across all sub-engines. *Warning:* This component inherently violates the Phase 15 frozen governance rule (forbidden combination of Promise + Dasha + Transit into a single score).
*   **`natal_promise_engine.py` (Active):** Calculates domain-specific promise (Marriage, Career, etc.). Depends on planet/house/karaka outputs. Outputs domain scores, grades, and detected afflictions.
*   **`planet_strength_engine.py` (Active):** Evaluates Shadbala and dignities. Depends on BAV/Varga engines. Outputs 9 planet objects with `final_score` and flags.
*   **`quality_metrics_engine.py` (Active):** Analytical pipeline tool. Generates histograms and discrimination metrics for test suites.
*   **`question_engine.py` (Active):** Keyword router and final response composer. Outputs aggregated dicts combining Promise and Timing.
*   **`rasi_strength_engine.py` (Active):** Evaluates 12 zodiac signs. Outputs `final_score` and flags per sign.
*   **`transit_engine.py` (Deprecated):** Legacy Gochara engine. Formally banished from the core probability engine during Phase 14G decoupling.
*   **`varga_engine.py` (Active):** Calculates D9/D10 dignity modifiers. Outputs vargottama bonuses and dignities.
*   **`yoga_engine.py` (Active):** Evaluates specific classical yogas. Outputs list of triggered yogas and maximum strengths.

## SECTION 2: FORMULA SYSTEM AUDIT
*(Location: `backend/app/formulas/`)*

*   **`registry_data.py`:** A pure data file containing exactly 44 `FormulaSchema` definitions (11 Base Families, 33 Variants).
*   **`loader.py`:** Contains `FormulaRepositoryLoader`. It resolves inheritance by fetching the `parent_formula_key` and appending its arrays (`required_signals`, `required_confidence_layers`, `required_engines`) onto the child Variant's arrays.
*   **`evaluator.py`:** Contains `FormulaEvaluator`. It executes the logic:
    1. Extracts only the `isolated_signals` required by the formula from the massive chart payload.
    2. Checks for Engine Degradation (if an engine failed to run).
    3. Runs Pure Boolean Gating: It counts how many `required_confidence_layers` are fulfilled. If 100% matched = `FAVORABLE`. If 0% = `UNFAVORABLE`. Otherwise = `MIXED`.
*   **`composer.py`:** Contains `AnswerComposer`. Uses the evaluator result to fetch a `.txt` template file and bundles the `isolated_signals` into a strict JSON string block for future LLM ingestion.

## SECTION 3: QUESTION ENGINE AUDIT
*(Location: `backend/app/engines/question_engine.py`)*

1.  **Input Contract:** `compose_response()` expects strictly segregated dictionaries: `question: str`, `domain: str`, `natal_promise: dict`, `dasha_activation: dict`, `transit_activation: dict`, `final_probability: dict`.
2.  **Output Contract:** Returns a JSON structure isolating the variables:
    ```json
    {
      "question": "str",
      "domain": "str",
      "routed": true,
      "probability": { "score": float, "grade": "str", "raw": float },
      "natal_promise": { "score": float, "promise": "str", "karaka": "str", "afflictions": [] },
      "timing": { "mahadasha": "str", "antardasha": "str", "pratyantardasha": "str", "dasha_strength": float, "activation_level": "str" },
      "transit": { "activation_score": float },
      "factor_breakdown": {},
      "answer_text": "str"
    }
    ```
3.  **Dependencies:** It consumes data generated by `NatalPromiseEngine`, `DashaEngine`, `TransitEngine` (passed in as stubs), and `MasterProbabilityEngine`.
4.  **Formula Flow:** `route_domain()` -> Returns domain string -> Orchestrator gets formula from `loader` -> `evaluator` evaluates -> `compose_response()` packs the final outputs.

## SECTION 4: ANSWER COMPOSER AUDIT
*(Location: `backend/app/formulas/composer.py`)*

1.  **Input JSON Schema:** Expects a `FormulaEvaluationResult` Pydantic object containing `final_state`, `isolated_signals` (dict), and `system_warnings`.
2.  **Output Schema:** Returns a `ComposerPromptPackage` Pydantic object containing:
    *   `prompt_template_id`
    *   `system_prompt` (Loaded from text files)
    *   `user_prompt` (Set to None)
    *   `evidence_block` (String)
3.  **Sample Evidence Block Output:**
    ```text
    FINAL STATE: FAVORABLE
    ASTROLOGICAL EVIDENCE:
    {
      "7th_house": { "final_score": 75.0 },
      "venus": { "final_score": 80.0 }
    }
    ```
4.  **Combination Logic:** The composer performs **NO mathematical logic**. It blindly takes whatever signals the `FormulaEvaluator` isolated and dumps them into a formatted string, relying 100% on the Evaluator's `final_state` to pick the correct system prompt.

## SECTION 5: MASTER PROBABILITY ENGINE
*(Location: `backend/app/engines/master_probability_engine.py`)*

*   **Purpose:** Aggregates Promise, Planet, House, Rasi, Varga, Dasha, and Transit into a single weighted mathematical score.
*   **Inputs:** `engine_outputs` dictionary containing the raw outputs of all 7 independent engines.
*   **Outputs:** A dictionary with a single `final_score` (0-100), `grade`, and `breakdown` array.
*   **Status:** **Legacy / Transitional Component.** As explicitly defined by the Governance Freeze, mixing Promise + Dasha + Transit into a single floating-point multiplier is now forbidden. The system has shifted to Formula-driven Boolean Gating. This engine remains for backwards compatibility with the V1 pipeline but is architecturally obsolete.

## SECTION 6: MANDALI READINESS CHECK

**Can a new standalone MandaliEngine be added without modifying the core engines?**
**YES.**

**Explanation:**
The `FormulaEvaluator` strictly extracts only the exact signals listed in the `FormulaSchema` in `registry_data.py`. Because all Transit and Mandali parameters were purged from these schemas during Phase 14G, the Question Engine is completely blind to transits. We can inject a new `MandaliEngine` into the orchestration pipeline, run it in parallel, and output its JSON directly to the LLM or UI without touching a single line of code in the `NatalPromiseEngine`, `DashaEngine`, or `FormulaEvaluator`.

## SECTION 7: RECOMMENDED PHASE 15 DESIGN

Based on the actual audited repository interfaces:

1.  **Recommended Location:** `backend/app/engines/mandali_engine.py`
2.  **Proposed Input Schema:**
    *   `current_ephemeris`: Dict of current planetary longitudes.
    *   `natal_moon_longitude`: Float (For padding/Gochara boundaries).
3.  **Proposed Output Schema:**
    ```json
    {
      "climate": "FAVORABLE",
      "active_mandalis": [4, 9],
      "monthly_trend_commentary": "Jupiter transiting 9th from Moon provides support.",
      "warnings": []
    }
    ```
4.  **Integration Point:** The new JSON output should be fed directly into `AnswerComposer.compose()` as an optional `mandali_advisory` block, appended below the `ASTROLOGICAL EVIDENCE` block. It must bypass the `FormulaEvaluator` entirely.
5.  **Risks:** If the `AnswerComposer` passes this to an LLM, the LLM may hallucinate that a "NEGATIVE" Mandali climate cancels out a "FAVORABLE" Dasha evaluation. Strict System Prompting must be enforced in the templates to mandate that the Formula's `FINAL STATE` is absolute law.
