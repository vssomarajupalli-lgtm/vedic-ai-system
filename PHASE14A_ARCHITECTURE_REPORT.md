# PHASE 14A: FORMULA LIBRARY EXPANSION ARCHITECTURE REPORT

## 1. Objectives Met
Successfully designed the architecture for expanding the Formula Repository from its initial seed state to full Question Registry coverage. The architecture prevents formula explosion and strict adherence to governance constraints was prioritized.

## 2. Key Architectural Decisions

### 2.1 Formula Scaling Strategy
As explicitly approved via the Architectural Challenge Requirement, the system adopts a **Formula Family + Inheritance Model**:
- **Master Category ➔ Subcategory ➔ Formula Family (Base) ➔ Formula Variant (Child) ➔ Question Mapping**
- This many-to-one mapping ensures that dozens of semantically similar question IDs share the same mathematical formula, preventing duplicate configurations.

### 2.2 Canonical Report Compatibility
Incorporated the mandatory **Evaluate Once, Consume Many** requirement.
- The `FormulaEvaluationResult` generated by the Formula Evaluator is legally declared as the single source of truth for the entire system.
- Downstream systems (LLM Answer Composer, Canonical PDF Report, Frontend dashboards) are strictly forbidden from recalculating astrology. They must visually map and render the exact deterministic boolean values and extracted signals contained in the payload.

### 2.3 Future Mandali Compatibility
Designed the Base formulas to support future Moon-centered Mandali transits natively via the `future_gochara_required` flag, requiring zero architectural engine redesigns when the Gochara phase is eventually executed.

## 3. Governance Audit
- **No hidden scoring:** Confirmed. Formulas scale linearly through boolean layers.
- **No duplicated astrology:** Confirmed via Domain-Specific duplication prevention and Base/Child Inheritance.
- **No evaluator complexity creep:** Confirmed. The Evaluator remains completely blind to inheritance; it processes flattened variants only.
- **No template logic creep:** Confirmed. Templates remain tied to overarching categories (`timing`, `strength`), not individual formulas.
- **No LLM decision making:** Confirmed.

Phase 14A design is complete. The system is structurally ready to scale the formula JSON files without risking unmaintainable bloat.
