# PHASE 9 STEP 1 – REPORT PROFILE FIX
**Date:** 2026-06-19

## Root Cause Found
The client profile metadata (`dob`, `tob`, `pob`, `latitude`, `longitude`, `timezone`) was being stripped from the pipeline data precisely at the parsing perimeter. The `JsonNormalizer._normalize_metadata` method was intentionally filtering the extraction dictionary down to only `name`, `ascendant_sign`, and `ascendant_degree`, discarding the rest of the `birth_data` object. 

As a result, the `pipeline_data` dictionary generated by `PipelineRunner.process()` arrived at the `ReportBuilder` completely devoid of birth information, forcing the reports to display `Unknown`.

## Files Modified
1. `backend/app/parsers/json_normalizer.py`: Added `dob`, `tob`, `pob`, `latitude`, `longitude`, and `timezone` to the returned `metadata` dictionary map. This passes the payload through the parsing firewall safely without impacting downstream astrology logic.
2. `backend/app/reports/schemas.py`: Replaced the loose `client_info: Dict` with a strict `ClientProfile` Pydantic BaseModel to enforce schema integrity inside `FinalReportSchema`.
3. `backend/app/reports/builder.py`: Updated `ReportBuilder` to construct the new `ClientProfile` block directly from the existing `pipeline_data.get("metadata")`, allowing a fallback to `machine_index.native_info` if necessary.
4. `backend/tests/test_report_builder.py`: Updated the assertion test logic to accommodate the new schema dot-notation (`report.client_profile.name`).
5. `backend/app/reports/templates/base.html`: Restructured the "Client Profile" UI grid to include all 6 required fields plus the generated timestamp. 

## Data Flow Correction
`canonical_content["birth_data"]` → `JsonNormalizer` (Passes fields) → `pipeline_data["metadata"]` → `ReportBuilder` → `FinalReportSchema.client_profile` → `HTMLGenerator` & `PDFGenerator`.
